  1. Product concept

drawi is a visual learning platform for tutors and students.

It has three connected product layers:

1 Live lessons

◦ A tutor and one or more students communicate using audio and video.

◦ Everyone works on a synchronized infinite canvas.

◦ The tutor can explain, observe student work, control permissions, create checkpoints, and guide the viewport.

2 Persistent boards

◦ A lesson does not disappear after the call ends.

◦ The board, checkpoints, metadata, and access rules remain available.

◦ Students can revisit the material.

◦ Tutors can convert selected material into reusable educational content.

3 Profiles and libraries

◦ Tutors can maintain public educational pages containing published boards and collections.

◦ Students have private learning libraries containing lesson history, saved materials, personal copies, and private notes.

◦ Every user may act as a tutor in one context and a student in another.

The long-term product direction is not merely “YouTube for boards.” It is:

A visual learning library where every lesson becomes structured, revisitable knowledge.

Place these in a server-only authorization module.

Every mutation service must:

2 Load the relevant database entity.

3 Run the central authorization function.

4 Validate the input.

5 Execute the mutation inside a transaction when needed.

6 Emit an audit event for sensitive actions.

7 Return a typed result.

  1. Board synchronization security

The Next.js application and sync worker are separate trust boundaries.

Implement short-lived board access tokens.

Recommended token claims:

type BoardAccessClaims =

sub: string;

boardId: string;

permission: "manage" | "edit" | "view";

lessonId?: string;

sessionId: string;

jti: string;

iat: number;

exp: number;

issuer: "drawi-web";

audience: "drawi-sync";

};

Prefer asymmetric signing:

• The web application signs with a private key.

• The sync worker verifies with a public key.

• The sync worker must not need the web application’s private key.

• Keys must be environment-specific.

• Tokens must expire quickly.

• Never put tokens into logs or URLs.

The sync worker must:

• Verify signature.

• Verify issuer.

• Verify audience.

• Verify expiration.

• Verify board ID.

• Verify required permission.

• Reject malformed claims.

• Prevent view-only users from mutating the board.

• Close or downgrade connections after permission revocation when feasible.

• Reject cross-board token reuse.

• Rate-limit repeated invalid connection attempts.

• Use structured redacted logs.

Dynamic lesson permission updates:

1 Tutor changes a student’s edit permission through the web application.

2 The server validates that the tutor can manage the lesson.

3 PostgreSQL is updated.

4 The server sends an authenticated internal permission update to the board Durable Object.

5 The Durable Object updates its enforced permission state.

6 The student interface receives a low-frequency control notification.

7 The student toolbar changes immediately.

8 The synchronization backend remains authoritative even if the student modifies the browser.

Do not implement student-edit disabling only by hiding tools.

2 Mark the lesson as ending.

4 Persist final lesson metadata.

7 Set endedAt.

8 Preserve tutor ownership.

9 Add or preserve tutor library item as owned.

  1. Engineering rules

    • TypeScript strict mode is mandatory.

    • Avoid any.

    • An isolated any requires a comment explaining why it is unavoidable.

    • Do not suppress TypeScript errors to finish faster.

    • Do not disable ESLint rules globally without an ADR-level reason.

    • Do not use non-null assertions for values that can genuinely be absent.

    • Use discriminated unions for state machines.

    • Use transactions for multi-record invariants.

    • Use database constraints.

    • Avoid giant components.

    • Avoid giant service files.

    • Avoid circular package dependencies.

    • Avoid generic utility dumping grounds.

    • Keep third-party APIs behind adapters.

    • Do not expose raw database records directly to Client Components.

    • Do not return secrets from Server Actions.

    • Do not log tokens.

    • Do not use fake production data.

    • Development seed data must be visibly fictional.

    • Do not add placeholder buttons.

    • Do not mark TODO functionality as complete.

    • Do not silently catch errors.

    • Do not use empty catch blocks.

    • Do not mutate unrelated code.

    • Do not perform destructive Git operations.

    • Do not use the tldraw demo server in production.

    • Do not send whiteboard strokes through LiveKit.

    • Do not publish lesson boards automatically.

    • Do not expose student history publicly.

    • Do not treat “saved” as “owned.”

    • Do not let a live lesson mutate a reusable tutor template.

    • Do not rely only on client-side permissions.

• Fresh clone installs using documented commands.

• Lockfile is committed.

• Environment variables are documented.

• Local database starts successfully.

• Database migrations apply from zero.

• Seed data runs only in development.

• Formatting check passes.

• Lint passes.

• Typecheck passes.

• Unit tests pass.

• Production build passes.

• Sync-worker build passes.

• Playwright critical tests pass.

• No unexpected browser console errors occur.

Authentication
